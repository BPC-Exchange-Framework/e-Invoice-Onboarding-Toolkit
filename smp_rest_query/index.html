    SMP REST API Query

    Query to an SMP to obtain terminal or final "Endpoint" destination of the e-invoice is implemented by two REST web service calls.

    See the BPC SMP Profile Version 1.0 (https://github.com/BPC-Exchange-Framework/BPC-Market-Pilot/blob/f3844411c40dd4b84276fd3ef6020d247afd83c4/BPC%20SMP%20Profile%20Version%201.0.pdf) document for some general guidance as to the form of this service call. See the OASIS website for the authoritative OASIS SMP 2.0 specification.

    Section 3 of the BPC SMP profile regarding the REST interfacace specifies in Section 3.1 that client authentication must not be required when accessing SMP [server] resources.

    The actual REST API for the webservice calls is referenced in section 3.2 of the BPC SMP profile, mandating the implementation as spcified in section 5.2 of the OASIS SMP 2.0 standard.

    !!!+ note "The BPC SMP Profile implements the OASIS SMP 2.0 Specification" Per section 3.2 of the BPC SMP profile, the BPC profile is an impelmentation of the OASIS SMP 2.0 standard.

    To the extent that section 6 of the BPC Profile has a requirement that the SMP client "must" validate the signature of the SMP [server response] as directed in section 5.6.2.2 of the OASIS SMP 2.0 standard, the OASIS standard is permissive and says the SMP client "may" verify the signature.

    While both the BPC SMP Profile and the OASIS SMP 2.0 standard are replete with non-functional requirements for an SMP client, less light is shed on the constructing the actual call. Section 5.4 of the OASIS SMP 2.0 standard most directly addresses the required REST call.

    The REST Webservice calls to the SMP service

    Call #1 to obtain ServiceGroup data

    The first webserivce call is construced of:

    1a. The webservice being queried with fully qualfied domain.

    The fully qualifiled web based SMP application uri with domain is the first element of the URL. In the development and test environment of the Onboarding Toolkit the uri or web address was "smp-api.sc-b2b.us" This value will be different in the Market Pilot.

    1b. The SMP Rest Binding

    Section 5.5 of the OASIS SMP 2.0 Standard directs the SMP REST Binding should be oasis-bdxr-smp-2. In the prior Section 5.4 "Resources" the exmaple URI indicates an SMP Binding of bdxr-smp-2. The development and test environment of the Onboarding Toolkit the SMP Binding used bdxr-smp-2.

    1c. The pre-hash URN

    The next element in the call to obtain the ServiceGroup data is the Specification, Schema, and PartyID used to create the URN for the SML NATPR DNS Query. The OASIS SMP 2.0 standard calls these the identifier schema and the participant id. When concatenated together, they form the pre-hash urn.

    1d. Create the string

    The ServiceGroup webservice call then takes the format of: https://\<webservice.domain>/\<smp_rest_binding>/\<urn> where the URN is \<specification>:\<schema>::\<party_id>.
    Using the terms employed by OASIS SMP 2.0 standard the URL is constructed as:
    https://\<webservice.domain>/\<smp_rest_binding>/\<identifier\ _scheme>::\<participant_id>

    #### 1e. Make the webservice call

    Example

    protocol = "https"
uri = "smp-api.sc-b2b.us"
standard = "bdxr-smp-2"
identifier_scheme = "urn:oasis:names:tc:ebcore:partyid-type:iso6523:0088"
participant_id = "123456789"
urn = identifier_scheme + "::" + participant_id 
query_url_1 = protocol + "://" + uri + ":" + identifier_scheme + "::" + participant_id
query_url_2 = protocol + "://" + uri + ":" + urn
print(f"Value of query_url_1: {query_url_1}")
print(f"Value of query_url_2: {query_url_2}")
print(f"constructed urls are the same: {query_url_1 == query_url_2}")

    Output

    Value of query_url_1: https://smp-api.sc-b2b.us:urn:oasis:names:tc:ebcore:partyid-type:iso6523:0088::123456789
Value of query_url_2: https://smp-api.sc-b2b.us:urn:oasis:names:tc:ebcore:partyid-type:iso6523:0088::123456789
constructed urls are the same: True
